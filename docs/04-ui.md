# Diseño UI/UX — MiniERP (AGENTIQ)

---

## 1. Objetivo de la interfaz

La interfaz del MiniERP debe comunicar control, orden y profesionalismo. Es una herramienta interna para que AGENTIQ administre clientes, proyectos, tareas y empleados. No es un sitio público ni una landing page; es un panel de gestión pensado para uso diario por perfiles técnicos y no técnicos.

### Prioridades de experiencia de usuario

1. **Claridad ante todo** — El usuario debe entender en segundos qué hay en pantalla, qué puede hacer y dónde hacer clic. Sin ambigüedad.
2. **Eficiencia en tareas frecuentes** — Crear proyectos, cambiar estados de tareas y consultar KPIs son acciones diarias. Deben requerir la menor cantidad de clics posible.
3. **Visibilidad del estado** — El dashboard, el Kanban y el Gantt existen para dar visibilidad. La interfaz debe priorizar la lectura de estado sobre la decoración.
4. **Prevención de errores** — Las acciones destructivas (eliminar) deben pedir confirmación. Los formularios deben validar antes de enviar. Los datos deben protegerse de entradas inválidas.
5. **No sorprender al usuario** — Las transiciones deben ser predecibles. Un clic en "Guardar" guarda. Un clic en "Eliminar" pide confirmación. No hay acciones silenciosas.

### Sensación deseada

| Atributo | Cómo se logra |
|---|---|
| Profesional | Paleta sobria, tipografía limpia, espaciado generoso, sin adornos innecesarios |
| Clara | Jerarquía visual fuerte, títulos visibles, contraste suficiente, etiquetas explícitas |
| Moderna | Bordes suaves, sombras sutiles, transiciones rápidas, diseño plano con jerarquía |
| Fácil de entender | Navegación lateral con íconos + texto, acciones visibles, breadcrumbs en detalle |
| Apropiada para gestión interna | Sin efectos llamativos, sin colores brillantes, sin animaciones decorativas |

## 2. Principios de diseño

### Claridad visual
- Cada pantalla tiene un título principal visible.
- Los datos se presentan en tablas o tarjetas, nunca en párrafos de texto.
- Los estados usan etiquetas de color (badges) con texto, no solo color.

### Consistencia
- Todos los formularios siguen el mismo patrón: etiqueta arriba, campo abajo, error debajo.
- Todos los botones de acción primaria usan el mismo color y posición.
- Los diálogos de confirmación tienen la misma estructura.
- Las tablas comparten columnas de acciones en la misma posición (derecha).

### Jerarquía
- Título de página → Subtítulo o descripción → Contenido principal → Acciones secundarias.
- Los botones primarios son visualmente dominantes. Los secundarios son outline o ghost.
- Las acciones destructivas (eliminar) son rojas y requieren dos clics (confirmación).

### Accesibilidad (mínima requerida)
- Navegación por teclado: todos los elementos interactivos deben ser accesibles vía Tab.
- Focus visible en todos los inputs, botones y enlaces.
- Los mensajes de error se asocian al campo mediante `aria-describedby`.
- Los botones tienen texto descriptivo (no íconos sin label oculto).
- Contraste mínimo de 4.5:1 para texto normal.
- Las tablas usan `<th>` y `scope` para ser comprensibles con lectores de pantalla.
- Los modales atrapan el foco y lo devuelven al cerrarse.
- Las notificaciones de estado (toasts) son anunciadas por el lector de pantalla.

### Prevención de errores
- Los botones de submit se deshabilitan mientras hay errores de validación.
- Los campos obligatorios tienen asterisco visible.
- La eliminación siempre requiere confirmación explícita.
- Las acciones con efectos irreversibles muestran un resumen de lo que se eliminará.

### Retroalimentación inmediata
- Toda acción del usuario produce una respuesta visual en menos de 300ms.
- Las operaciones asíncronas muestran un estado de carga (spinner o skeleton).
- Las mutaciones exitosas muestran un toast de confirmación.
- Las mutaciones fallidas muestran un toast con el error legible.

### Responsive design
- La interfaz debe ser funcional en mobile, tablet y desktop.
- El sidebar se colapsa en mobile (hamburger menu).
- Las tablas se vuelven scroll horizontal o cambian a vista de tarjetas.
- El Kanban se convierte en lista vertical en mobile.

### Reducción de carga cognitiva
- Los formularios largos se agrupan en secciones lógicas.
- Los selectores cargan opciones de la base de datos en lugar de requerir escritura manual.
- Los filtros y búsquedas usan debounce para evitar operaciones innecesarias.
- Los estados vacíos ofrecen una acción clara ("Crear primer cliente").

## 3. Arquitectura de información

### Navegación principal

La navegación es un menú lateral fijo (sidebar) con ícono + texto para cada módulo. El sidebar permanece visible en desktop. En mobile se colapsa y se despliega mediante un botón hamburguesa.

```
┌─────────────┬──────────────────────────────────────────────┐
│   Sidebar   │              Main Content                     │
│             │                                               │
│  [Logo]     │   [PageHeader: título + breadcrumbs + acc.]   │
│             │                                               │
│  ■ Inicio   │   ┌──────────────────────────────────────┐   │
│  ■ Clientes │   │                                      │   │
│  ■ Proyectos│   │        Contenido de la página         │   │
│  ■ Tareas   │   │                                      │   │
│  ■ Empleados│   └──────────────────────────────────────┘   │
│             │                                               │
│             │                                               │
└─────────────┴──────────────────────────────────────────────┘
```

### Orden de navegación

1. **Inicio** — Dashboard con KPIs (primera pantalla al cargar la aplicación).
2. **Clientes** — Gestión de clientes.
3. **Proyectos** — Gestión de proyectos + acceso al Gantt.
4. **Tareas** — Tablero Kanban global (tareas de todos los proyectos).
5. **Empleados** — Gestión de empleados.

### Encabezado (Header)

Contenido del header dentro del layout:

- Breadcrumbs de navegación (en páginas de detalle y formularios).
- Sin usuario/logout (no hay autenticación).
- Sin selector de tema (modo oscuro fuera de alcance).

### Breadcrumbs

Se implementan en páginas de detalle y formularios donde la profundidad de navegación lo justifica:

| Ruta | Breadcrumbs |
|---|---|
| `/proyectos` | Inicio > Proyectos |
| `/proyectos/nuevo` | Inicio > Proyectos > Nuevo proyecto |
| `/proyectos/[id]` | Inicio > Proyectos > [Nombre del proyecto] |
| `/proyectos/gantt` | Inicio > Proyectos > Cronograma |

No se usan breadcrumbs en la página de inicio ni en listados principales (el sidebar ya indica posición).

### Acciones globales

No hay acciones globales en el header. Cada página tiene sus acciones en el PageHeader (ej: "+ Nuevo cliente").

### Comportamiento en móvil

- Sidebar oculto por defecto. Se abre mediante botón hamburguesa en la esquina superior izquierda.
- Los PageHeaders se apilan verticalmente (título arriba, acciones abajo).
- Los breadcrumbs se acortan con "..." si exceden el ancho disponible.

## 4. Inventario de pantallas

### Dashboard

| Aspecto | Detalle |
|---|---|
| **Objetivo** | Dar lectura rápida del estado general de la empresa mediante KPIs y resúmenes visuales |
| **Ruta** | `/` |
| **Tipo de componente** | Server Component (lecturas), con secciones Client Component (gráfico opcional) |

**Información principal:**
- Grilla de 6 tarjetas de KPI (StatCard): Proyectos activos, Proyectos completados, Tareas pendientes, Tareas vencidas, Total de clientes, Empleados activos.
- Listado de "Próximos vencimientos": top 5 tareas con fecha límite más cercana.
- Gráfico opcional de proyectos agrupados por estado (barras horizontales o dona). Se implementa solo después de completar los KPIs obligatorios (ver prioridad en sección 16).

**Prioridad de implementación del Dashboard:**
1. KPIs numéricos obligatorios (6 tarjetas).
2. Lista de próximos vencimientos.
3. Gráfico de proyectos por estado — solo si queda tiempo después de los puntos 1 y 2.

**Acciones disponibles:**
- Ninguna acción directa en los KPIs (son informativos).
- Clic en un ítem de "Próximos vencimientos" redirige al detalle de la tarea o proyecto.
- Clic en una tarjeta de KPI podría llevar al listado correspondiente (mejora opcional).

**Estados:**
| Estado | Comportamiento |
|---|---|
| Carga | Skeleton de 6 rectángulos del tamaño de las StatCards + skeleton de gráfico + 5 filas de skeleton en lista |
| Vacío | KPIs en cero con valor "0". Lista de vencimientos vacía ("No hay tareas con fecha límite próxima"). Gráfico sin datos (mensaje "No hay proyectos registrados") si está implementado |

## 5. Flujos principales

### 5.1 Crear cliente

1. Usuario navega a Clientes → "Nuevo cliente".
2. Llena el formulario (nombre obligatorio, email obligatorio, otros opcionales).
3. Hace clic en "Guardar".
4. El formulario valida en cliente. Si hay errores, se muestran debajo de cada campo.
5. Se envía POST a `/api/clientes`. El servidor valida con Zod.
6. Éxito: Toast verde "Cliente creado correctamente". Redirección a `/clientes`.
7. Error (email duplicado): Toast rojo "Ya existe un cliente con ese email".

### 5.2 Crear proyecto

1. Usuario navega a Proyectos → "Nuevo proyecto".
2. Selecciona cliente del select (cargado desde BD).
3. Llena nombre, fechas, prioridad.
4. Selecciona empleados del multi-select (empleados activos cargados desde BD).
5. Hace clic en "Guardar".
6. Validación: fecha_entrega > fecha_inicio. Al menos un empleado (sugerido).
7. POST a `/api/proyectos` con los IDs de empleados seleccionados.
8. El servidor crea el proyecto + inserts en proyecto_empleado en una transacción.
9. Éxito: Toast + redirección a `/proyectos/[id]`.

### 5.3 Asignar empleados a proyecto

1. Desde crear/editar proyecto, el multi-select muestra todos los empleados activos.
2. Los empleados ya asignados aparecen preseleccionados.
3. Al guardar, el servidor sincroniza la tabla proyecto_empleado:
   - Elimina registros que ya no están seleccionados.
   - Inserta registros nuevos.
   - Mantiene los existentes.
4. Si un empleado con tareas en el proyecto es removido, sus tareas quedan con `empleado_id = NULL` (manejo en aplicación).

### 5.4 Crear tarea

1. Desde detalle del proyecto o Kanban, clic en "+ Nueva tarea".
2. El proyecto ya está precargado (readonly).
3. Selecciona empleado (solo los asignados al proyecto aparecen en el select).
4. Llena título, prioridad, fecha límite.
5. POST a `/api/tareas`. Validación RN-08 si hay empleado seleccionado.
6. Éxito: Toast + redirección al Kanban o detalle.

### 5.5 Cambiar estado de tarea

**Opción base (sin drag-and-drop):**
1. Desde el Kanban, cada tarjeta tiene botones "←" (retroceder estado) y "→" (avanzar estado).
2. Alternativa: select desplegable con los tres estados.
3. PUT a `/api/tareas/[id]` con el nuevo estado.
4. El servidor ejecuta `revalidatePath()` para refrescar el Kanban.
5. Éxito: Toast sutil o animación de la tarjeta moviéndose de columna.

**Reglas de estado:**
- Pendiente → En progreso (permitido).
- En progreso → Completada (permitido).
- No se permite retroceder (Completada → En progreso) - RN-13.

### 5.6 Editar registro

1. Desde listado o detalle, clic en "Editar" (lápiz).
2. Se carga formulario precargado con datos existentes.
3. Usuario modifica y hace clic en "Guardar".
4. PUT a API. Validación Zod.
5. Éxito: Toast + redirección al detalle o listado.

### 5.7 Eliminar registro

1. Desde listado o detalle, clic en "Eliminar" (papelera).
2. Se abre ConfirmDialog con mensaje específico según entidad.
3. Si hay impedimentos (cliente con proyectos), el mensaje lo indica y el botón está deshabilitado.
4. Usuario confirma → DELETE a API.
5. El servidor maneja ON DELETE constraints.
6. Éxito: Toast + redirección al listado.
7. Error (RESTRICT triggers): Toast con mensaje de error.

### 5.8 Consultar dashboard

1. Al cargar la aplicación o navegar a Inicio, el Server Component consulta Supabase con consultas agregadas.
2. Las StatCards se renderizan en servidor con valores numéricos: Proyectos activos, Proyectos completados, Tareas pendientes, Tareas vencidas, Total de clientes, Empleados activos.
3. La lista de próximos vencimientos se renderiza a partir de los datos disponibles.
4. El gráfico opcional se renderiza solo si está implementado.
5. El usuario ve toda la información sin necesidad de interacción.
6. Para refrescar, recarga la página (o se refresca automáticamente después de mutaciones via `revalidatePath('/')`).

## 7. Tipografía

### Familia tipográfica

**Inter** (sans-serif) para todo el sistema. Es una fuente moderna, legible en pantalla, con buena altura de x y pesos bien definidos. Se carga desde Google Fonts con `display=swap`.

- Kanban: scroll horizontal entre columnas.
- Tablas: scroll horizontal si es necesario.

### Comportamiento en móvil (<768px)

- Sidebar oculto, menú hamburguesa.
- Dashboard: KPIs en grid de 2 columnas, luego 1 columna si es necesario.
- Tablas: mostrar columnas esenciales solamente (ocultar secundarias en móvil) o cambiar a vista de tarjetas.

## 10. Formularios

### Estándares

**Etiquetas:**
- Arriba del campo, alineadas a la izquierda.
- Peso medium (500), tamaño 14px.
- Color: gray-700.

**Campos obligatorios:**
- Marcar con asterisco rojo `*` junto al label.
- No es necesario incluir texto "(*) Campo obligatorio" si todos los campos obligatorios están marcados consistentemente.
- Opcional: texto "Todos los campos son obligatorios salvo que se indique lo contrario" al inicio del formulario.

**Mensajes de error:**
- Debajo del campo correspondiente, en texto rojo de 12px.
- Deben ser específicos: "El nombre es obligatorio", "El email no tiene un formato válido", "La fecha de entrega debe ser posterior a la fecha de inicio".
- Aparecen después del primer blur o después de intentar submit (modo `onSubmit` de React Hook Form).

**Ayudas de campo:**
- Texto gris de 12px debajo del campo, solo si es necesario aclarar el formato esperado.
- Ejemplo: "Ej: 123456789-0" para el NIT.

**Selectores (select nativo o shadcn Select):**
- Los selects cargan opciones desde la base de datos vía Server Component.
- El placeholder debe ser una instrucción: "Seleccionar cliente", "Seleccionar empleado".
- No incluir opción en blanco seleccionable (usar placeholder).

**Fechas (date picker):**
- Input de tipo `date` nativo de HTML5, con icono de calendario.
- Validación adicional: fecha_entrega > fecha_inicio.
- En edición, mostrar la fecha existente precargada.

**Estados de carga durante envío:**
- Botón "Guardar" muestra spinner y texto cambia a "Guardando...".
- Todos los inputs se deshabilitan (disabled).
- No permitir clic en "Cancelar" (se puede mantener habilitado, pero se ignora si hay envío en curso).

**Botones:**
- Botón primario (Guardar): color de fondo azul primario, texto blanco. En móvil: ancho completo.
- Botón secundario (Cancelar): estilo outline o ghost, gris. Sin borde si es ghost.
- Ambos botones al final del formulario, alineados a la derecha en desktop, apilados en móvil (Cancelar arriba, Guardar abajo, o viceversa).

**Cancelación:**
- Vuelve a la página anterior sin confirmación (no hay datos sin guardar si se usa React Hook Form con `shouldUnregister`).
- Si hay datos modificados sin guardar (form dirty), mostrar ConfirmDialog: "¿Descartar cambios?".

**Prevención de doble envío:**
- Botón deshabilitado después del primer clic.
- El estado de envío (`isSubmitting` de React Hook Form) controla esto.

**Confirmación de éxito:**
- Toast de éxito después de la mutación.
- Redirección automática a la página de listado o detalle correspondiente.

| **Alternativa accesible** | Vista tabular: tabla de proyectos ordenable por fecha inicio/fecha entrega, con duración calculada en días. Sirve como vista alternativa cuando el Gantt no es práctico (pantalla pequeña) o como fallback |
| **Complejidad** | Media |
| **Prioridad** | Alta (requisito funcional del brief) |

**Decisión sobre librería Gantt:** Usar Frappe Gantt (MIT, liviana, SVG-based) envuelta en un Client Component dentro del módulo Proyectos. Si la integración resulta problemática, implementar barras con CSS puro posicionadas según diferencia de fechas.

## 13. Accesibilidad

### Requisitos mínimos

**Navegación por teclado:**
- Todos los botones, enlaces, inputs y selects deben ser accesibles mediante Tab.
- El orden de tabulación debe seguir el orden visual (izquierda a derecha, arriba a abajo).
- El sidebar debe tener atajo de teclado (opcional, mejora).

**Focus visible:**
- Todos los elementos interactivos deben tener un anillo de focus visible (`ring-2 ring-blue-500`).
- No usar `outline: none` sin proporcionar un reemplazo visible.

**Etiquetas accesibles:**
- Todos los inputs deben tener un `<label>` asociado (no placeholder como única identificación).
- Los botones con solo ícono deben tener `aria-label`.
- Los enlaces deben tener texto descriptivo.

**Contraste:**
- Texto normal (14px+): ratio de contraste mínimo 4.5:1.
- Texto grande (18px+ o 14px bold): ratio mínimo 3:1.

**Mensajes de error:**
- Asociados al campo mediante `aria-describedby`.
- Anunciados por lectores de pantalla con `role="alert"`.

**Botones con nombre accesible:**
- `aria-label="Editar cliente"` en botones de ícono.
- `aria-label="Eliminar proyecto"` en botones de papelera.

**Tablas comprensibles:**
- Usar `<thead>`, `<th>` con `scope="col"`.
- Para tablas con datos complejos, agregar `aria-label` descriptivo.

**Modales accesibles:**
- Enfocar el primer elemento interactivo al abrirse.
- Atrapar el foco dentro del modal (focus trap).
- Cerrar con tecla Escape.
- Devolver el foco al elemento que activó el modal al cerrarse.

**No depender únicamente del color:**
- Los estados usan badge con texto, no solo color.
- Las barras de progreso incluyen texto del porcentaje.
- Los iconos de acciones incluyen texto en tooltip o aria-label.

**Soporte para prefers-reduced-motion:**
- Deshabilitar animaciones de skeleton (pulso) cuando `prefers-reduced-motion: reduce`.
- Transiciones de estado deben ser instantáneas en ese caso.

- **Error global:** `error.tsx` con mensaje "Algo salió mal. Intenta de nuevo." + botón "Reintentar".
- **Error 404:** `not-found.tsx` con "Página no encontrada" + link "Volver al inicio".
- **Error de recurso (API 404):** Toast "El recurso solicitado no existe o fue eliminado".
- **Error de validación (API 400):** Toast con el mensaje de error del servidor.
- **Error de constraint (API 409):** Toast "No se puede eliminar: el registro tiene dependencias".
- **Error de red/TimeOut (API 503):** Toast "Error de conexión. Verifica tu conexión a internet e intenta de nuevo."

### Success

- **Crear:** Toast verde "[Entidad] creado correctamente." + redirección.
- **Actualizar:** Toast verde "[Entidad] actualizado correctamente." + redirección.
- **Eliminar:** Toast verde "[Entidad] eliminado correctamente." + redirección.
- **Cambio de estado:** Toast verde "Estado actualizado a [Nuevo estado]." (sin redirección, permanece en Kanban).

### Disabled

- Botón de submit deshabilitado cuando hay errores de validación o mientras se envía.
- Inputs deshabilitados durante el envío del formulario.
- Checkbox/radio en readonly (no aplica en este proyecto).
- Botón "Eliminar" en ConfirmDialog deshabilitado si hay impedimentos (cliente con proyectos).

### Saving

- Botón primario muestra spinner + texto "Guardando...".
- Todos los inputs del formulario se deshabilitan.
- El botón "Cancelar" permanece visible pero se ignora si se hace clic (o se deshabilita también).

### Deleting

- Botón "Eliminar" en ConfirmDialog muestra spinner + texto "Eliminando...".
- El diálogo no se puede cerrar mientras se elimina.

### Offline / Fallo de conexión

- **Consideración:** El sistema está diseñado para funcionar siempre online (consulta a Supabase). No hay modo offline.
- Si hay fallo de red al cargar una página, el `error.tsx` muestra el mensaje genérico.
- Si hay fallo de red al enviar un formulario, se muestra Toast de error y los datos del formulario se conservan (React HookForm mantiene el estado).
- No hay service workers, ni caché offline, ni IndexedDB en este alcance.

## 17. Riesgos de UX

| ID | Riesgo | Impacto | Mitigación |
|---|---|---|---|
| UX-01 | **Tablas demasiado anchas** con 7+ columnas en pantallas medianas | Medio — el usuario debe hacer scroll horizontal constante | Priorizar columnas esenciales; ocultar columnas secundarias en breakpoints menores con clases `hidden md:table-cell` |
| UX-02 | **Formularios largos** (proyecto tiene 10 campos + multi-select) | Medio — el usuario puede sentirse abrumado o cometer errores | Agrupar campos en 3 secciones visuales (Datos generales, Fechas, Asignación). Separar con títulos de sección |
| UX-03 | **Exceso de acciones** visibles por fila (editar + eliminar + más) | Bajo — confusión visual, ruido en la interfaz | Limitar a 2 acciones por fila (editar, eliminar). Si se necesitan más, agrupar en menú de tres puntos |
| UX-04 | **Kanban difícil en móvil** (3 columnas no caben) | Alto — la funcionalidad core de tareas queda inutilizable en mobile | Migrar a vista de lista con filtro por estado en móvil. Los botones de cambio de estado siguen funcionando |
| UX-05 | **Gantt ilegible** en pantallas pequeñas | Alto — el cronograma pierde su propósito si no se ve | Scroll horizontal con indicador visual. Asegurar que las etiquetas de proyecto sean legibles. Alternativa: acordeón por estado en móvil |
| UX-06 | **Eliminaciones accidentales** sin suficiente confirmación | Alto — pérdida de datos irreparable | Siempre usar ConfirmDialog con mensaje explícito. Para proyectos, incluir advertencia de tareas y asignaciones afectadas. Botón rojo como señal de peligro |
| UX-07 | **Estados poco claros** — el usuario no sabe si una tarea está pendiente, en progreso o completada | Medio — confusiones en el seguimiento | Badges con color + texto siempre visible. Evitar usar solo color para indicar estado. Tooltip con descripción del estado si es necesario |
| UX-08 | **Proyecto sin empleados asignados** — luego no se pueden crear tareas | Medio — el usuario no entiende por qué no puede asignar empleados a una tarea | En el formulario de tarea, si no hay empleados asignados al proyecto, mostrar mensaje informativo: "Este proyecto no tiene empleados asignados. Asigna empleados desde la edición del proyecto" |
| UX-09 | **Retroalimentación lenta** después de mutaciones (el usuario no sabe si su acción funcionó) | Medio — ansiedad del usuario, posibilidad de clics repetidos | Mostrar estado "Guardando..." inmediatamente en el botón. Toast de éxito/error siempre. `revalidatePath()` asegura que los datos se refresquen |
| UX-10 | **Multi-select de empleados sin indicación de carga** | Bajo — el usuario no sabe si hay más empleados disponibles | El select debe cargar opciones con un skeleton o spinner mientras se obtienen los datos. Indicar cantidad: "5 empleados disponibles" |

## 18. Criterios de aceptación UI/UX

El diseño UI/UX se considera aprobado cuando:

1. **Navegación:** Todos los módulos son accesibles desde el sidebar en máximo 2 clics.
2. **Dashboard:** Las 6 tarjetas de KPI se renderizan correctamente con valores numéricos. El gráfico y la lista de vencimientos se muestran o muestran estado vacío.
3. **CRUD Clientes:** Se puede crear, listar (con búsqueda), ver detalle, editar y eliminar un cliente. La eliminación con proyectos asociados muestra advertencia.
4. **CRUD Proyectos:** Se puede crear (con selección de cliente y empleados), listar (con % avance), ver detalle (con equipo y tareas), editar y eliminar. El Gantt muestra todos los proyectos en una línea de tiempo.
5. **CRUD Tareas:** Se puede crear (con asignación a empleados del proyecto), editar, eliminar y cambiar estado. El Kanban muestra 3 columnas con tarjetas y permite cambiar estado.
6. **CRUD Empleados:** Se puede crear, listar (con filtros), ver detalle (con carga de trabajo), editar y eliminar.
7. **Validaciones:** Todos los formularios validan campos obligatorios. Fechas inválidas muestran error. Emails duplicados muestran error.
8. **Estados:** Las pantallas muestran estado de carga (skeleton), vacío (EmptyState) y error (ErrorState) según corresponda.
9. **Confirmaciones:** Toda acción de eliminación muestra ConfirmDialog. Las acciones destructivas tienen botón rojo.
10. **Retroalimentación:** Después de cada mutación, se muestra un toast de éxito o error. Los botones de submit muestran estado de carga.
11. **Responsive:** La interfaz es funcional en desktop (1280px+), tablet (768px) y móvil (375px). No hay contenido cortado ni superposición.
12. **Teclado:** Todos los elementos interactivos son accesibles mediante Tab. El focus es visible. Los modales atrapan el foco.
13. **Contraste:** El texto cumple con ratio de contraste mínimo 4.5:1. Los estados son distinguibles sin color.
14. **Coherencia:** Todos los formularios siguen el mismo patrón. Todas las tablas comparten estilo. Todos los diálogos de confirmación tienen la misma estructura.
